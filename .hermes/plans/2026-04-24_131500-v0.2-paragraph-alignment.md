# Nivitz-doc-diff — v0.2.0: Paragraph-Level Alignment Fix

## Problem

The v0.1.0 algorithm aligned paragraphs strictly by index (old[i] vs. new[i]). If a paragraph was **inserted** or **deleted** in the middle of the document, every paragraph after it shifted by one position, and those shifted paragraphs got incorrectly diffed against unrelated neighbors. The result: huge spurious "replacements" that look like entire paragraphs were rewritten when in fact only a title was added.

Real-world example from the user's academic paper: adding one title line before a paragraph caused the tool to interleave title words with paragraph body words, producing gibberish like `"LearningThe vs.effect Minimal-Movesof Goalsgoal"`.

## Solution: Two-Level Diff

Apply `SequenceMatcher` **twice** — once at the paragraph level, then once within each paragraph at the word level.

### Level 1: Paragraph alignment

Treat each paragraph's text as a single "token". Run `SequenceMatcher` on the two lists of paragraph texts. This gives opcodes that tell us which paragraphs are equal, which are inserted, which are deleted, and which are replaced.

Critically, SequenceMatcher handles shifts: if a paragraph was inserted at position 3, paragraphs 0, 1, 2 align with 0, 1, 2 (equal), new paragraph 3 is a pure insert, and old paragraphs 3, 4, 5... still align with new paragraphs 4, 5, 6... (equal).

### Level 2: Word-level diff within matched/replaced paragraphs

For paragraphs marked `equal` at level 1 → copy unchanged.
For paragraphs marked `insert` at level 1 → output highlighted.
For paragraphs marked `delete` at level 1 → output strikethrough.
For paragraphs marked `replace` at level 1 → this is the interesting case. A `replace` block can contain *multiple* old and new paragraphs. We need a secondary alignment within the block before diffing word-by-word, otherwise we re-create the original bug at a smaller scale. Strategy:
- If the block is 1 old vs. 1 new: do word-level diff between them (existing behavior).
- If the block is N old vs. M new: for each old paragraph, find its best match among the new paragraphs using a similarity ratio (`SequenceMatcher.ratio()`). Only pair them if the ratio exceeds a threshold (e.g. 0.4). Unmatched olds → delete. Unmatched news → insert. Matched pairs → word-level diff.

### Similarity Threshold

Word-level diff only makes sense when paragraphs are *related*. If two paragraphs share <40% of their words, treating one as a rewrite of the other just produces noise. Better to render them as one full delete + one full insert so the reader can see cleanly "this paragraph was removed, this one was added."

Threshold: 0.4 (tunable). Choose based on `SequenceMatcher.ratio()` over word tokens.

## Implementation Plan

### Changes to `diff_engine.py`

Add three things:

1. **`DocDiffOp`** dataclass — represents a paragraph-level operation:
   - `kind`: `"equal" | "insert" | "delete" | "replace"`
   - `old_paragraph`: `ParagraphData | None`
   - `new_paragraph`: `ParagraphData | None`
   - `word_ops`: `list[DiffOp]` — word-level diff ops (only populated for equal/replace when the pair is a real match; empty for pure insert/delete)

2. **`compute_paragraph_alignment(old, new)`** — returns `list[DocDiffOp]` representing paragraph-level alignment.

3. **Rewrite `compute_diff`** to use paragraph alignment first, then word-level diff within matches. Output is now a list of `DocDiffOp` rather than `list[list[DiffOp]]`.

### Changes to `docx_writer.py`

Rewrite `build_diff_document` to consume the new `DocDiffOp` structure. For each paragraph-level op:
- `equal`: add a paragraph with unchanged text (or if `word_ops` has changes, render those — equal at paragraph level from the outer match can still have word-level tweaks if we decide to diff equal-matched paragraphs; in v0.2 we keep it simple: `equal` = paragraph text matched exactly).
- `insert`: add a paragraph with all text highlighted.
- `delete`: add a paragraph with all text strikethrough.
- `replace` with matched word_ops: render each word_op (equal/insert/delete/replace) per v0.1 logic.

Actually, the existing `DocDiffOp.word_ops` is populated for matched pairs only. Cleanest: if `kind == "equal"` AND `word_ops` is empty, just output plain text; if `word_ops` non-empty (e.g., whitespace-only differences), render them.

Simpler model: always populate `word_ops` for equal/replace when both old and new paragraphs exist.

### Test Coverage

New tests in `test_diff_engine.py`:
1. Title insertion before paragraph — paragraph body remains unchanged
2. Paragraph deletion in the middle — surrounding paragraphs align correctly
3. Multiple consecutive insertions — all marked as inserts
4. Reordering (paragraph moved) — should detect via similarity matching
5. Unrelated replacement block — two dissimilar paragraphs should render as delete+insert, not word-level merge

## Risks

- **Performance**: SequenceMatcher on paragraph text (each "token" is a whole paragraph's text) can be slow for huge documents. For reasonable essay/paper sizes (~100 paragraphs) this is fine. Concern flagged, not addressed in v0.2.
- **Similarity threshold 0.4**: arbitrary. Could be wrong for some documents. Expose as `--similarity-threshold` CLI arg in v0.3 if needed.
- **Reordering**: true paragraph reordering will be treated as delete+insert, not a "move." Detecting moves is a known hard problem in text diffing, deferred.
